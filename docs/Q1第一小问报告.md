# Q1 第一小问报告：规则框架与随机AI

## 一、代码框架设计

本项目的整体框架参考了课程提供的围棋参考代码（`reference/go/`），沿用了其核心架构：`GameState` 不可变设计（每次走子返回新状态）、统一的 AI 接口（`select_move(game_state) -> Move`）、以及数据层→规则层→状态层→AI 层的分层结构。这套设计在围棋框架中已经验证可行，直接移植到象棋可以减少基础设施的工作量，专注于象棋规则本身。

与围棋框架的主要差异有两处。一是棋盘表示：围棋用 `Point -> GoString` 追踪棋串和气，象棋只需 `Point -> Piece` 存储单个棋子，结构更简单。二是围棋框架保留了完整的历史状态链用于劫判定，象棋无此需求，去掉了这部分开销。

此外，Q2 的搜索 AI 对 `GameState` 做了一项关键改造：在 Alpha-Beta 搜索中，每个节点需要频繁地走子和撤销，若每次都 `deepcopy` 棋盘，性能开销极大。为此在 `Board` 上增加了 `make_move() / unmake_move()` 接口，搜索时直接在原棋盘上原地修改并撤销，避免了拷贝。`GameState.apply_move()` 仍保留 `deepcopy` 语义供对弈流程使用，两套接口并存，互不干扰。

## 二、规则设计思路

走法生成与合法性检查分两层处理。`ccmoves.py` 中每种棋子对应一个独立函数，只负责生成该棋子在物理上能到达的目标格，不考虑游戏层面的约束。`GameState.legal_moves()` 在此基础上再调用 `_is_move_legal()` 过滤掉走后将帅照面或己方将被将军的走法。这样棋子规则和游戏规则各自独立，便于分别测试。

三类特殊规则的处理思路各有不同。马蹩腿和象塞眼本质上是"路径阻断"：马走日字时先检查正交方向的紧邻格是否有子，有子则整个方向跳过；象走田字时检查中心格，逻辑完全对称。炮则不同，它的移动和吃子是两套规则：移动时与车相同（遇子停），吃子时必须跳过恰好一个棋子（炮架），因此炮的走法生成分两个阶段，先收集移动目标，遇到第一个子后切换到"找吃子目标"模式，继续扫描直到遇到第二个子。

合法性检查复用了 `make_move() / unmake_move()`：模拟走子后检查将帅是否照面、己方将是否暴露，再撤销恢复棋盘，不需要额外的棋盘拷贝，与搜索中的零拷贝逻辑共用同一套接口。

随机 AI 本身没有特殊逻辑，从 `legal_moves()` 返回的列表中均匀随机选一个走法，其作用是驱动完整的对弈循环，验证规则框架在实际运行中的正确性。
