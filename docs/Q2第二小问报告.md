# 第二小问：搜索AI的设计与实现

## 一、算法选择与整体思路

我们实现了三种搜索算法，各有不同的定位：

- **Alpha-Beta 剪枝**（主力）：象棋的子力价值和位置优劣相对容易量化，能写出靠谱的评估函数，因此确定性的穷举搜索比统计采样更合适。Alpha-Beta 在有限时间内尽可能深地搜索博弈树，配合评估函数给出确定性的判断。
- **Minimax**（对照组）：Alpha-Beta 的"去剪枝版本"，代码几乎相同，唯一区别是不维护搜索窗口、不做剪枝。用于精确衡量剪枝本身的贡献。
- **MCTS**（跨范式对照）：基于采样而非穷举的搜索范式。MCTS 的优势场景是评估函数难以设计的博弈（如围棋），在象棋中天然受限于分支因子大（~40）、模拟收敛慢的问题，但作为跨范式对照仍有实验价值。

三种算法共享同一个评估函数 `evaluate()`，使用相同的 5 秒时间限制，确保对比实验中变量可控。

---

## 二、Alpha-Beta：关键设计决策

教科书上的 Alpha-Beta 算法本身并不复杂，真正决定棋力的是工程实现中的一系列决策。以下是我们做出的、不同于朴素实现的关键选择。

### 2.1 Negamax 形式

传统 Minimax 需要分别处理"最大化玩家"和"最小化玩家"两套逻辑。我们采用 Negamax 形式，利用零和博弈的对称性将两套逻辑统一：

```
score = -negamax(child, depth-1, -beta, -alpha)
```

每次递归时交换玩家身份并取反分数和窗口，一套代码处理所有情况。

### 2.2 零拷贝搜索

这是性能提升最大的单一决策。

朴素实现中，每次尝试一步棋都需要 `deepcopy` 整个棋盘状态，搜索树中每个节点都产生一次完整拷贝。我们改为在棋盘上原地修改（`make_move`），搜索完成后撤销（`unmake_move`）：

```python
captured = board.make_move(move)    # 原地执行，返回被吃棋子
score = -negamax(board, ...)        # 在同一个棋盘上递归
board.unmake_move(move, captured)   # 撤销，恢复原状
```

效果：搜索深度从 3 层提升到 4-5 层，这对棋力的影响是决定性的。

### 2.3 迭代加深与时间控制

我们不固定搜索深度，而是从深度 1 开始逐层加深，在时间预算内尽可能深搜：

```
for depth = 1, 2, 3, ...:
    if 已用时间 > 80% 时间限制: 停止
    best_move = search(depth)
```

这样做有两个好处：一是总能在时间限制内返回一个合法走法（即使只搜了 1 层）；二是浅层搜索的结果可以为深层搜索提供走法排序的参考。

### 2.4 走法排序

Alpha-Beta 剪枝的效率高度依赖搜索顺序——如果先搜到好棋，后续差棋就能被快速剪掉。我们采用两层排序策略：

- **吃子走法**：按 MVV-LVA（最有价值受害者 - 最低价值攻击者）排序。用兵吃车排在最前面，因为这类走法几乎总是好棋。
- **非吃子走法**：按历史启发排序。记录过去搜索中导致剪枝的走法，下次优先尝试。计数器按 `depth²` 加权，深层剪枝的走法权重更高。

### 2.5 置换表

同一个局面可能通过不同的走法顺序到达。置换表缓存已搜索局面的结果，避免重复计算。每个条目记录搜索深度、分数、标志类型（精确值/上界/下界）和最佳走法。

## 三、Minimax：刻意的简化

Minimax 的实现与 Alpha-Beta 几乎完全相同——同样的 Negamax 框架、同样的零拷贝搜索、同样的迭代加深和走法排序。唯一的区别是：**不维护 alpha/beta 窗口，不做剪枝**，遍历每个节点的所有子节点。

这种"刻意的简化"是为了在 Q3 对比实验中提供干净的对照组：两个算法共享所有工程优化，唯一的差异就是"有没有剪枝"，从而精确衡量剪枝本身的贡献。

由于没有剪枝，Minimax 在相同时间内只能搜到深度 2-3，而 Alpha-Beta 能搜到 4-5。

## 四、MCTS：不同于经典实现的决策

MCTS 的四个步骤（选择、扩展、模拟、反向传播）是标准的，但我们在模拟策略和性能优化上做了几个关键的非标准决策。

### 4.1 混合模拟策略

经典 MCTS 的模拟阶段是纯随机走子直到终局。这在象棋中不可行：象棋的分支因子约 40，一局可能超过 200 步，纯随机模拟一局需要数百毫秒，5 秒内只能完成几十次模拟，远不足以建立有效的搜索树。

我们的做法是**随机走 5 步后直接用评估函数打分**：

```
模拟(state):
    在 state 上随机走 5 步（偏好吃子走法）
    return tanh(evaluate(state) / 1000)
```

这样做有两个考虑：
1. 5 步随机走子引入了局面的随机扰动，保留了蒙特卡洛采样的本质特征；
2. 使用与 Alpha-Beta 相同的 `evaluate()` 函数，确保对比实验中评估函数不是变量。

### 4.2 零拷贝模拟

模拟阶段是 MCTS 中调用最频繁的部分。最初我们使用 `GameState.apply_move()`（内部做深拷贝），5 秒内只能完成约 177 次模拟。改为直接在棋盘上 `make_move/unmake_move` 后，模拟次数提升到约 85,000 次——提升约 600 倍。

### 4.3 吃子偏好

模拟中的随机走子并非完全均匀随机。我们以 70% 的概率优先选择吃子走法，使模拟更接近真实对局的走子分布，提高模拟结果的信息量。

### 4.4 合法走法缓存

每个树节点在首次访问时生成并缓存合法走法列表，后续的 `is_fully_expanded` 和 `get_unexplored_move` 直接查缓存，避免重复调用开销较大的走法生成器。

## 五、评估函数

三种算法（除 Random 外）共享同一个评估函数，采用 **MATL + LOC** 双特征体系：

- **MATL（子力价值）**：为每种棋子赋予数值权重（车 > 炮 ≈ 马 > 象 > 士 > 卒），过河卒额外加分。
- **LOC（位置价值）**：为每种棋子在棋盘上的每个位置定义价值表，鼓励棋子占据有利位置（如巡河车、过河卒等）。

子力权重通过**自适应遗传算法（AGA）**自动调优，避免人工调参的主观性。AGA 的核心思想是：优秀个体降低交叉率和变异率以保留优良基因，较差个体提高变异率以增加探索。经过 50 代进化，马的权重从 400 提升到 523（+30.8%），象从 200 提升到 315（+57.5%），反映了这两种棋子在实战中被低估的机动性价值。

## 六、总结

三种算法形成了一个有层次的对比体系：

| 算法 | 搜索范式 | 5秒内搜索规模 | 搜索深度 |
|------|---------|-------------|---------|
| Alpha-Beta | 穷举 + 剪枝 | ~5,000 节点 | 4-5 层 |
| Minimax | 穷举（无剪枝） | ~3,000 节点 | 2-3 层 |
| MCTS | 采样 | ~85,000 次模拟 | 不适用 |

Alpha-Beta 是主力算法，通过零拷贝搜索、走法排序、置换表等工程优化，在 5 秒时间限制内达到深度 4-5，具备中等棋力。Minimax 和 MCTS 作为对照组，将在第三小问的对比实验中与 Alpha-Beta 进行系统性比较。
